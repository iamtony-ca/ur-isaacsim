# cuMotion robot config (UR16e + 2F-85)

GPU motion planning for the UR16e via **NVIDIA Isaac ROS cuMotion** as a MoveIt
planning pipeline. cuMotion plans on the GPU; execution stays on our existing
`scaled_joint_trajectory_controller` (sim or real).

## Files
- `ur16e_2f85.xrdf` — cuMotion robot config (cspace, collision spheres, self-collision)
  for UR16e + GRP-ES-CPL-077 coupling + 2F-85. Generated by `gen_xrdf.py`.
- `ur16e_2f85.urdf` — flattened UR16e+2F-85 URDF with **absolute `file://` mesh paths**
  + `gripper_frame`/`grasp_frame` (cuMotion tool/grasp frames). What the cuMotion
  node loads. (Mesh paths point at `/opt/ros/jazzy/share/...`; regenerate if your
  install prefix differs.)
- `gen_xrdf.py` — regenerates the XRDF from the URDF + the shipped ur10e template.

## How it was made (regeneration)
UR16e shares shoulder/wrist/base + the whole 2F-85 collision meshes with UR10e, so
we start from NVIDIA's hand-tuned `ur10e_robotiq_2f_85.xrdf` and only regenerate the
links whose mesh differs/are new: `upper_arm_link`, `forearm_link`,
`ur_to_robotiq_link` (coupling). Spheres are generated from each link's collision
mesh with the standalone cuMotion engine and transformed into the link frame.

```bash
# 1) cuMotion engine wheel (offline sphere tool) — needs python3.12-venv
cd /isaac-sim/ur_ws/deps                      # cumotion-1.1.0-cuda13.0-x86_64.tar.gz extracted here
python3 -m venv .venv-cumotion
./.venv-cumotion/bin/pip install \
    cumotion-1.1.0-cuda13.0-x86_64/python_wheels/cumotion-1.1.0-cp312-cp312-linux_x86_64.whl \
    trimesh pyyaml numpy

# 2) regenerate the cuMotion URDF (flatten xacro, package:// -> file://, add grasp frames)
cd /isaac-sim/ur_ws && source install/setup.bash
xacro src/ur_bringup/urdf/ur16e_2f85/ur16e_2f85_sim.urdf.xacro \
  | sed 's#package://#file:///opt/ros/jazzy/share/#g' > deps/cumotion_gen/ur16e_2f85.urdf
#   (then append gripper_frame/grasp_frame — see deps/cumotion_gen/, or copy from this dir)

# 3) regenerate the XRDF
./deps/.venv-cumotion/bin/python deps/cumotion_gen/gen_xrdf.py   # -> ur16e_2f85.xrdf
cp deps/cumotion_gen/ur16e_2f85.{urdf,xrdf} src/ur_bringup/cumotion/
```

## Run (sim)
```bash
ros2 launch ur_bringup ur16e_2f85.launch.py                              # control (or _d405 / real)
ros2 launch ur_bringup ur16e_2f85_d405_cumotion_moveit.launch.py        # move_group + cuMotion (RViz)
python3 src/ur_bringup/isaac/common/moveit_plan_execute_demo.py          # programmatic plan+execute
```
- **`ur_only` (default true)** — move_group/RViz use a UR-arm-only (6-joint) model +
  plain ur SRDF so **interactive RViz** cuMotion works (the plugin doesn't filter the
  request's full start_state; RViz sends the whole robot state, so a 12-joint start is
  rejected against the 6-joint c-space). cuMotion still plans WITH the gripper
  (collision spheres come from this XRDF). `ur_only:=false` keeps the full
  gripper-aware move_group but then only **programmatic** cuMotion works (empty
  start_state, e.g. the demo); interactive RViz `<random valid>` will fail.
- In RViz: planner = `isaac_ros_cumotion` (default; Context tab) -> Goal State /
  marker -> **Plan -> Execute**.
- **Reset / recover pose**: repeated planning can drift a wrist past +/-2*pi -> MoveIt
  then rejects all plans ("start state out of bounds"). Recover with:
  `python3 src/ur_bringup/isaac/common/reset_pose.py home`  (direct-to-controller,
  works even out of bounds; also `up` / `zero`). RViz named states (home/up) work
  too once in bounds.

## Real-time obstacle avoidance (world collision) — DONE (sim)
cuMotion only does self/scene collision until you feed it a world ESDF. This is
now wired with **isaac_ros_nvblox** + **robot_segmenter**:

    ros2 launch ur_bringup ur16e_2f85_d405_nvblox.launch.py            # segmenter + nvblox + static-cam TF
    ros2 launch ur_bringup ur16e_2f85_d405_cumotion_moveit.launch.py read_esdf_world:=true

Pipeline: `static workspace camera depth -> robot_segmenter (mask robot) -> nvblox
(3D ESDF, base_link) -> cuMotion` (cuMotion reads `/nvblox_node/get_esdf_and_gradient`).

**Key lesson:** map with a STATIC workspace camera, NOT the eye-in-hand D405 — the
wrist camera mostly sees the robot and moves with it, polluting the TSDF so the
start pose is reported in world collision. The D405 stays grasp/vision-only.
Config `config/ur16e_2f85_d405/nvblox_cumotion.yaml` (nvblox `global_frame` MUST
equal cuMotion's XRDF base frame = `base_link`). Full write-up + gotchas:
`../../HARDWARE.md` §4, `../../HISTORY.md` §12.

See `../../HARDWARE.md` §4 for the Isaac ROS cuMotion install (apt repos, CUDA 13,
VPI) and the ABI/JTC gotchas.
